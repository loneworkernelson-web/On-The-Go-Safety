Part 1: Security Keys — The Golden Rule

🔐 Save Your Keys — Right Now

Your system uses two secret keys generated in Factory Step 1. These must be saved somewhere secure (e.g. a password manager) before you proceed with deployment.

KeyPurposeWho Needs It
Secret Key (Admin)Authenticates the backend script. Used to access the spreadsheet and unlock the Monitor.Administrators only — never share.
Worker Key (App)Authenticates worker devices connecting to the backend.Embedded in config.json — workers don't enter it manually.

⚠️ If you deploy with different keys later: Every device will lose its connection and require a full reinstall. If you use the same keys in future config updates, devices update seamlessly in the background. There is no penalty for keeping the same keys indefinitely.

Where to find your keys: Open the OTG Factory app (index.html) — your keys are shown in Step 1 → Security Connection Keys. The Factory remembers them between sessions using your browser's local storage. They are not re-printed in this document for security reasons.

Configure (Factory Step 1)

In the Factory app, fill in all fields on the Config tab:

  • Organisation Name — appears in all emails and generated documents
  • App Icon Label — short name shown under the home screen icon (max 12 characters)
  • Security Keys — click 🔄 to auto-generate both keys, or enter your own. The Secret Key must be 32–64 characters (the counter turns green when valid).
  • Regional Logic — sets the timezone, voice locale, and vehicle roadworthy term (%%VEHICLE_TERM%% for NZ, Rego for AU, etc.)
  • App Branding — upload your logo; the Factory will extract your brand colours and preview the app live
  • Safety & Fleet Options — configure alarm grace period, mileage tracking, travel reports, welfare check-ins, vehicle tracking, AI redaction, API keys, and notification services
Save your keys now — before moving to Step 2. See Part 1 of this manual.
2

Database Setup (Factory Step 2)

You need a Google Sheet to store visit records, staff, sites, and report templates.

Option A — Recommended: Click "✨ Create Database Copy". This opens a pre-configured Google Sheet template in your browser. Make a copy to your own Google Drive.

Option B — Manual: Create a new sheet at sheets.new, create 4 tabs named exactly: Visits, Staff, Templates, Sites, then click each "Copy Blueprint" button and paste into the first row of each tab.

Identity Matching (important): The name in the Staff tab must match exactly what workers type into the setup wizard — case-sensitive.
3

Install the Backend (Factory Step 3)

The backend is a Google Apps Script that handles all safety logic — overdue detection, email alerts, SMS, AI report processing, and data archiving.

A. Paste the Code

  1. In the Factory, click "📋 Copy System Code".
  2. In your Google Sheet, go to Extensions → Apps Script.
  3. Delete all existing code in the editor.
  4. Paste and click Save (disk icon).

B. Deploy as a Web App

  1. Click Deploy (blue button) → New Deployment.
  2. Click the gear icon ⚙ next to "Select type" → choose Web App.
  3. Set Execute as: Me.
  4. Set Who has access: Anyone.
  5. Click Deploy. Accept the permissions prompt.
  6. Copy the Web App URL — you'll need it in Step 3 of the Factory.

C. Set Up Triggers (Automated Tasks)

Triggers are the automation engine — without them, overdue detection and alerting will not work. Set up all five:

A. Safety Watchdog Required
checkOverdueVisits
⏱ Time-driven · Every 10 minutes

Scans for workers who haven't checked out or confirmed safety within the grace period. Fires email + SMS alerts.

B. Privacy Shredder Required
cleanupPrivateSentNotes
⏱ Time-driven · Every hour

Removes raw personal notes from processed visit records after AI has processed them.

C. Daily Health Report Recommended
sendHealthEmail
⏱ Day timer · e.g. 7am–8am

Emails a daily system summary — visit counts, escalations, failed alert emails, any stale open visits. Your early-warning system.

D. Data Archiver Optional
archiveOldData
⏱ Week timer · e.g. Sunday

Moves completed visits older than 30 days to an Archive tab. Only closed visits are moved — active records are never touched.

E. Weekly Safety Summary Optional
sendWeeklySummary
⏱ Week timer · e.g. Monday 7am–8am

Emails a brief weekly digest — total visits, distance travelled, and safety alerts fired in the past 7 days. Useful for management reporting without opening the spreadsheet.

How to add triggers: In the Apps Script editor, click the Alarm Clock icon in the left sidebar → + Add Trigger (bottom right). Set the function name, source (Time-driven), and interval as shown above for each trigger.

D. Paste the Web App URL Back into the Factory

Return to the Factory, go to Step 3, paste your Web App URL into the URL field, then click "📡 Test System Connection". A green "System Link Established" response confirms everything is working before you proceed.

Deploy the Worker App (Factory Step 5)

A. Assemble Your Deployment Folder

Create a folder on your computer with all of the following files at the same level — no subfolders:

your-org-safety/ ├── index.html ← worker app (from OTG repository) ├── sw.js ← downloaded from Factory Step 4 ├── manifest.json ← PWA manifest (from OTG repository) ├── config.json ← your org config (downloaded from Step 4) ✦ ├── icon.png ← your logo, 512×512px recommended ✦ └── worker_guide.html ← worker guide (downloaded from Step 4) ◆ ✦ org-specific — everything else is shared across all deployments ◆ optional but recommended — linked automatically in the app's Resource Library

B. Upload to a Static Host

Drag your folder to Netlify Drop (app.netlify.com/drop) or push to a GitHub Pages branch.

Important — Netlify: After the initial upload, click "Sign up to claim site" so the URL doesn't expire after 24 hours. For future config updates, drag the new folder onto your existing Netlify site dashboard (not the Drop page) to update it in place without changing the URL.

C. Test Offline Before Launch

Install the PWA on a test device, load it once on Wi-Fi, then switch to Airplane Mode and reopen. Confirm it loads fully with correct branding. This verifies the offline-first safety guarantee.

D. Generate Your Onboarding Pack

Back in the Factory, go to Step 5 → Staff Onboarding Pack. Paste your deployed URL to generate:

  • A ready-to-send welcome email for staff with installation instructions
  • A QR code linking directly to the app (right-click to save as image)

Part 5: Updates & Configuration Changes

Performing a Config-Only Update (Recommended)

Under the Config Model, most changes — branding, intervals, feature toggles — only require replacing config.json. The worker app detects the updated version field and refreshes automatically.

1

Open the Factory

Open index.html (the Factory app). Your previous settings should be remembered in the browser's local storage. If not, re-enter them — especially your keys.

⚠️ Use your original keys. If you click 🔄 to regenerate new keys, devices will lose connection. Only use the same keys you originally deployed with.
2

Adjust Settings

Change whatever needs updating — new branding colour, different grace period, toggle a feature on or off, new API key, etc.

3

Download & Deploy New config.json

Go to Factory Step 4, click "⬇ Download config.json". Upload the new file to your Netlify site (drag the folder onto your existing site's dashboard — not the Netlify Drop page). The version stamp in the file changes automatically, which triggers devices to refresh their cached config next time they connect.

4

Backend Changes (if any)

If you also changed backend settings (API keys, escalation logic), you need to re-copy the Code.gs and deploy a new version in Apps Script. See the "New Version Trap" warning below.

⚠️ The "New Version" Trap — Apps Script

When you edit code in Google Apps Script, clicking Save does not update the live system. For any change to take effect:

  1. Click Deploy (blue button).
  2. Select Manage Deployments.
  3. Click the Pencil icon (Edit).
  4. Change "Version" to "New version".
  5. Click Deploy.

Modifying the Backend Logic (Advanced)

You can edit Code.gs directly to change system behaviour — email formatting, AI prompts, alert thresholds, etc.

  • Go to your Google Sheet → Extensions → Apps Script
  • Find the function to change (e.g. smartScribe for AI, sendAlertEmail for notifications)
  • Test with the Run button before deploying
  • Always follow the "New Version" deployment steps after saving changes

Part 6: Advanced Features

ntfy Push Notifications

ntfy is a free, open-source push notification service that delivers instant alerts directly to contacts' phones the moment an alarm fires — no email delay, no polling required.

  • Setup: Enter your ntfy server URL in Factory Step 1 (leave blank to use the free hosted ntfy.sh service). For higher privacy, self-host ntfy using Docker and paste your server URL.
  • How contacts subscribe: When an alarm fires, contacts automatically receive a subscribe link in the alert email. They install the free ntfy app and subscribe to their assigned topic.
  • Worker setup: During the app setup wizard, workers enter their contacts' ntfy topic names (provided by your administrator).

Healthchecks.io Dead-Man's Switch

The backend's escalation engine runs every ~15 minutes via a trigger. If that trigger silently stops firing (a known Google Apps Script limitation), workers could reach overdue status without any alarm. Healthchecks.io catches this.

  • Sign up free at healthchecks.io
  • Create one check with a 20-minute period and 15-minute grace
  • Copy the ping URL and paste it into the Healthchecks.io Ping URL field in Factory Step 1
  • The backend pings this URL after every successful escalation check. If it goes silent for ~35 minutes, Healthchecks.io emails you — before any worker reaches overdue status
Strongly recommended for any organisation where worker safety is critical. The free tier supports up to 20 checks.

Daily Health Report Email

When the sendHealthEmail trigger is configured (Trigger C), the system emails a daily summary each morning containing:

  • Total visits logged in the past 24 hours
  • Number of escalations fired
  • Any alert emails that failed to send
  • Workers with an open visit older than 24 hours (stale records)

Configure the recipient in Factory Step 1 (System Health Report Email). If left blank, the report goes to the Google account that owns the Apps Script. You can also trigger it manually at any time via OTG Admin → Send Health Email Now in the spreadsheet.

AI Privacy Redaction

When enabled in Factory Step 1, the AI report processor automatically removes worker names and other identifying information from visit notes before sending them to the Gemini AI API. This is a data minimisation practice and is strongly recommended when using any external AI service to process staff records.

You can toggle this on or off per deployment via the AI Privacy Redaction checkbox in Factory Step 1 → Safety & Fleet Options.

Part 7: Troubleshooting

SymptomLikely CauseFix
"Access Denied" when worker submits a report Worker Key mismatch — the key in config.json doesn't match the deployed backend Regenerate config.json using your original keys and re-upload. If you used new keys, you'll need to redeploy the backend with matching keys too.
"Failed to Fetch" on the Worker App Incorrect or outdated Web App URL in config.json Check your Apps Script deployment URL, update it in the Factory, regenerate and redeploy config.json.
Code changes ignored — backend behaviour unchanged The "New Version" trap — you saved but didn't deploy Deploy → Manage Deployments → Edit → New version → Deploy.
Monitor plays no sound Browser autoplay policy — audio requires user interaction first Click anywhere on the Monitor page before alerts are expected, then click Start Audio.
App shows wrong branding after config update Old config.json is cached — version bump may not have propagated yet Confirm the new config.json is uploaded. Wait 60 seconds and reopen the app. If still cached, open the app and leave it open — the service worker will fetch the new config in the background.
Overdue alerts not firing The Safety Watchdog trigger (Trigger A) is not running Check Apps Script → Triggers. Confirm checkOverdueVisits exists and is set to run every 10 minutes. Check the Executions log for errors.
Worker can't install PWA on iPhone Not using Safari — Chrome on iOS doesn't support "Add to Home Screen" for PWAs Worker must use Safari on iPhone. Share → Add to Home Screen.
Worker Guide / Ops Manual download button is greyed out Template files not present in the Factory folder Ensure guide_template.html and ops_manual_template.html are in the same folder as the Factory's index.html and refresh the Factory page.
Healthchecks.io is alerting — backend has gone silent Apps Script trigger quota exceeded, project error, or Google service disruption Check Apps Script Executions log. Re-create the trigger if needed. Check Google Workspace Status Dashboard for outages.
