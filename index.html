2. Set the Watchdog Alarms (Triggers)

Follow these steps exactly to automate your safety checks:

A. Safety Watchdog

  1. Click the Alarm Clock Icon (Triggers) on the sidebar.
  2. Click + Add Trigger (bottom right).
  3. Function: checkOverdueVisits.
  4. Source: Time-driven.
  5. Type: Minutes timer | Every 10 minutes.

B. Privacy Shredder

  1. Add a second trigger.
  2. Choose cleanupPrivateSentNotes.
  3. Source: Time-driven.
  4. Type: Hour timer | Every Hour.
  5. Click Save.

C. Daily Health Report

Recommended

This trigger emails a daily summary to your nominated health report address — visit counts, escalations fired, any failed alert emails, and workers with stale open visits. It is your early-warning system for silent backend failures.

  1. Add a third trigger.
  2. Choose sendHealthEmail.
  3. Source: Time-driven.
  4. Type: Day timer | choose a morning hour (e.g. 7am–8am).
  5. Click Save.

📧 Report recipient: Set in the System Health Report Email field on the Config tab. If left blank, the report goes to the Google account that owns the Apps Script. You can also run this manually at any time via OTG Admin > Send Health Email Now.

D. Data Archiver

Optional

Moves completed visits older than 30 days from the Visits tab into an Archive tab, keeping your live sheet fast and clean. Only closed visits (DEPARTED / SAFE / COMPLETED) are moved — active records are never touched.

  1. Add a fourth trigger.
  2. Choose archiveOldData.
  3. Source: Time-driven.
  4. Type: Week timer | choose a low-traffic day (e.g. Sunday).
  5. Click Save.

E. Weekly Safety Summary

Optional

Emails a brief weekly digest to the script owner — total visits, distance travelled, and number of safety alerts fired in the past 7 days. Useful for management reporting without opening the spreadsheet.

  1. Add a fifth trigger.
  2. Choose sendWeeklySummary.
  3. Source: Time-driven.
  4. Type: Week timer | choose a morning (e.g. Monday, 7am–8am).
  5. Click Save.

Deploy the Worker App

                


                

                    Under the Config Model the worker app is a single canonical codebase shared by all organisations. Only your config.json is org-specific. You update config, not the app.
                


                

                    

                        
1

                        

                            
Assemble Your Deployment Folder

                            
Create a folder on your computer and place all of the following files in it — everything at the same level, no subfolders.

                            

                                
your-org-safety/

                                

                                    
index.html — worker app (from OTG repository)

                                    
sw.js — service worker (from OTG repository)

                                    
manifest.json — PWA manifest fallback (from OTG repository)

                                    
config.json — your org config (downloaded from Step 4) ✦

                                

                                
✦ only org-specific file — logo, theme, and keys are all embedded inside it

                                
icon.png and manifest.json are only needed as fallbacks — the app generates both at runtime from config.json

                            

                        

                    


                    

                        
2

                        

                            
One-Drag Deployment (Netlify / GitHub Pages)

                            
Drag your folder to Netlify Drop or push to a GitHub Pages branch. The folder must contain index.html, sw.js, manifest.json, config.json, and icon.png.

                        

                    


                    

                        
3

                        

                            
Updating Config in Future

                            
To change settings (colours, intervals, keys), regenerate config.json here and replace the file on your host. Bump the version field — the app detects the change and refreshes automatically. No app rebuild needed.

                        

                    


                    

                        
4

                        

                            
Offline First — Test Before Launch

                            
Install the PWA on a test device, load once on Wi-Fi (caches all assets including config.json), then go to Airplane Mode and reopen. Confirm the app loads fully with correct branding. This verifies your offline-first safety guarantee.
